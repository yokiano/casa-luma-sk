# TEMP Plan: Neon/Postgres-backed Notion read cache

_Date: 2026-06-07 · last updated: 2026-06-08_

> **Ephemeral implementation artifact.** This file belongs under `temp/notion-cache/` and must be deleted after the Notion cache implementation is complete, merged, and documented in persistent docs. Keep only the durable docs/runbook that explain the final shipped behavior.

## Goal

Reduce repeated Notion API reads and improve page/server load times without adding Redis or another deployment.

The cache must be low-maintenance, safe around generated Notion SDK files, and easy to disable if anything behaves unexpectedly.

## Recommendation

Use **Neon/Postgres as the persistent cache store** via the existing project database connection.

Preferred library: **Keyv + `@keyv/postgres`**.

Why this over a custom table/class:

- `Keyv` gives a small standard cache API: `get`, `set`, `delete`, TTL, optional namespaces.
- `@keyv/postgres` handles the Postgres storage table and TTL metadata.
- Avoids manually maintaining a Redis-like cache class.
- Avoids adding Redis/Upstash/deployment maintenance.
- Works for moderate traffic, which is the stated use case.

## Important generated-SDK constraint

Some files under `src/lib/notion-sdk/**` are generated and may be overwritten by `pnpm notion:generate`.

Therefore, do **not** put hand-maintained cache logic directly inside generated database files.

Two acceptable approaches, in order of preference:

1. **If `src/lib/notion-sdk/core/src/generic-db.ts` is stable project-owned core:** add only a tiny hook there that delegates to app-owned cache utilities.
2. **If that core file is also regenerated:** do not edit it. Instead cache at app-owned call sites/facades, starting with high-impact public/server loaders.

Before implementation, confirm whether `src/lib/notion-sdk/core/src/generic-db.ts` is regenerated by `pnpm notion:generate`.

## Existing ad-hoc caches to unify

Do not leave parallel cache systems running forever.

| Location | Current behavior | Migration note |
| --- | --- | --- |
| `src/lib/server/website-images.ts` | In-process memory cache + request coalescing, 5 min TTL, bypassed in `dev` | Replace with Neon cache in a later integration task; remove module-level `cachedImages` once verified |
| Other server loaders | Mostly uncached Notion reads | Integrate incrementally via task packages below |

Document every replacement in the runbook so future debugging knows which layer served a response.

## Architecture

Create app-owned cache utilities outside the generated SDK, for example:

- `src/lib/server/cache/keyv.ts`
- `src/lib/server/cache/notion-cache.ts`
- `src/lib/server/cache/notion-cache-keys.ts`
- `src/lib/server/cache/README.md` (short pointer)
- `docs/notion/notion-cache-runbook.md` (persistent primary debugging doc — see Documentation section)

Responsibilities:

- Build stable cache keys from:
  - Notion operation name: `query`, `getPage`, `blockChildren`, etc.
  - database/page/block id
  - normalized request body/query parameters
  - optional projected/filter properties
  - cache namespace/version
- Store raw Notion API JSON responses so existing DTO/type parsing remains unchanged.
- Return the same response shape the Notion SDK already returns.
- Add TTL per usage category.
- Fail open: if cache read/write fails, log/report lightly and fetch from Notion normally.
- Emit structured cache events for debugging (hit/miss/disabled/error/bypass) — see Observability.

## Type-safety requirement

Do not create parallel DTO classes or manually mirror Notion response types.

Cache helpers should be generic and preserve input/output types:

```ts
async function cachedNotionRead<T>(options: {
  key: string;
  ttlMs: number;
  fetcher: () => Promise<T>;
}): Promise<T>
```

The cache stores serialized JSON, but the TypeScript type comes from the original Notion SDK call or generated database method.

## Cache scope

Cache only read operations:

- ✅ database/query reads
- ✅ page metadata reads
- ✅ block children reads, if used for public pages/content
- ❌ create/update/archive writes
- ❌ mutation-heavy workflows unless explicitly safe
- ❌ sensitive admin searches by default until reviewed

## TTL policy

Start conservative:

| Data kind | Suggested TTL |
| --- | ---: |
| public menu/content/images/events | 5–30 minutes |
| semi-admin lists/searches | 30–120 seconds |
| page metadata used for automation decisions | 15–60 seconds or no cache |
| writes/mutations | no cache |

Add per-call overrides instead of one global TTL.

## Invalidation policy

Minimal first version:

- TTL handles normal staleness.
- After app-originated Notion writes, delete related obvious keys when practical.
- Do not attempt complete perfect invalidation on day one.
- Add a global env kill-switch so stale-data problems can be solved instantly.

Suggested env vars:

- `NOTION_CACHE_ENABLED=true|false`
- `NOTION_CACHE_DEFAULT_TTL_SECONDS=300`
- `NOTION_CACHE_NAMESPACE=casa-luma:notion:v1`
- `NOTION_CACHE_DEBUG=true|false` (verbose structured logs; off in prod by default)

## Database / Neon considerations

Use the existing Postgres/Neon connection settings where possible.

Best practices:

- Reuse existing database connection env selection: `DATABASE_URL`, `POSTGRES_URL`, etc. (same order as `src/lib/server/db/client.ts`).
- Keep cache table separate/namespaced, e.g. table `notion_cache` or Keyv schema/table.
- Prefer a **separate Keyv/postgres client** from Drizzle's `db` proxy unless we confirm pool sharing is safe — cache traffic should not starve app queries.
- Add cleanup for expired rows if the adapter does not do enough automatically.
- Avoid caching huge payloads indefinitely.
- Use bounded TTLs; no permanent cache entries.
- Consider an unlogged table only if acceptable; default logged table is safer.

# Test strategy

Tests are **not optional** for this project area. Cache bugs are subtle; tests are the safety net.

## Unit tests (required, always run in CI)

File: `src/lib/server/cache/notion-cache.test.ts`

| Case | Assert |
| --- | --- |
| Cache hit | `fetcher` not called; returned value matches stored JSON |
| Cache miss | `fetcher` called once; result stored |
| Disabled (`NOTION_CACHE_ENABLED=false`) | `fetcher` always called; no store writes |
| Kill-switch off + env missing | Sensible default (document: default enabled or disabled — pick conservative) |
| Cache read error | Fail open → `fetcher` called; error logged |
| Cache write error | Fail open → return `fetcher` result anyway |
| TTL expiry | After TTL, `fetcher` called again (use fake timers or short TTL) |
| Concurrent miss coalescing | Two parallel calls with same key → single `fetcher` invocation (if implemented) |

File: `src/lib/server/cache/notion-cache-keys.test.ts`

| Case | Assert |
| --- | --- |
| Key stability | Same logical query → identical key |
| Key ordering | Object key order in query does not change key |
| Namespace prefix | Changing `NOTION_CACHE_NAMESPACE` changes key |
| Operation separation | Same id, different op (`query` vs `getPage`) → different keys |

## Integration tests (optional locally, recommended before merge)

Gate behind `NOTION_CACHE_INTEGRATION_TEST=1` and a real `DATABASE_URL`.

| Case | Assert |
| --- | --- |
| Postgres round-trip | `set` → `get` survives module reload |
| Expiry | Row not returned after TTL (adapter-dependent) |
| Table isolation | Cache table does not collide with Drizzle app tables |

## Target-level tests (per T4/T5)

For each integrated loader/facade:

| Case | Assert |
| --- | --- |
| Loader uses cache | Mock/spy on underlying Notion DB — second call avoids network |
| Response shape unchanged | Snapshot or structural assert on DTO mapping |
| Dev bypass policy | Document if `dev` skips cache (website-images today skips in dev — decide globally) |

## Manual verification checklist (before merge)

- [ ] Load public page twice; logs show `miss` then `hit` (with debug on).
- [ ] Set `NOTION_CACHE_ENABLED=false`; both loads show `disabled` / fetch each time.
- [ ] Change Notion content; within TTL content may be stale (expected); after TTL refreshes.
- [ ] After invalidation path (if implemented), content updates without waiting full TTL.
- [ ] Kill-switch works in production-like env without redeploy (env only).

# Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Stale Notion content | Conservative TTLs, env kill-switch, delete obvious keys after writes |
| Generated SDK overwrites cache code | Keep real logic outside generated files; confirm generation behavior first |
| Cache table growth | TTLs + expired-row cleanup |
| Type drift / duplicated DTOs | Cache raw response JSON and preserve generic return types |
| Cache outage breaks pages | Fail open and fetch from Notion |
| Over-caching admin/mutation workflows | Cache read-only public/high-repeat paths first |
| Connection pool exhaustion | Separate small pool or limits for Keyv; monitor Neon connections |
| Multi-session context loss | progress.md + handoff.md + concept_queue |
| Debug difficulty | Runbook + structured logs + integrated surfaces table |
