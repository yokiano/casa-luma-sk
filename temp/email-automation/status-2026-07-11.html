Email automation implementation status

This update adds modular Telegram notification templates, seeds default DB classification rules (mirroring the built-in matchers) with a dummy_input column, and gives the dashboard rule management: enable/disable, priority reorder, per-rule Telegram preview, and a Send-test button that posts a demo message to the production Telegram chat. What changed in this update

  • Modular notifications: the large renderEmailAutomationNotification was split into src/lib/server/email-automation/notifications/ with one template per kind (expenseRecorded, actionReady, reviewNeeded, ignored) in templates.ts, a dispatcher in render.ts, a sender in send.ts, shared helpers in helpers.ts, and built-in dummy inputs in builtin-dummies.ts. Templates stay code-backed (no DB templates).
  • Dashboard rule controls: enable/disable and up/down priority reorder for DB rules, with per-rule Telegram preview rendered from each rule's dummy_input.
  • Per-rule preview + Send test: every DB rule and every built-in classifier shows its Telegram message and has a Send-test button that posts a real demo message to the production Telegram chat.
  • dummy_input column + seed: migration 0005 adds email_classification_rules.dummy_input (JSONB) and seeds 8 default rules from seed-rules.ts, each with a sample input derived from the real .eml fixtures in temp/emails-for-automations/.
  • Deprecation intention: built-in matchers in builtInClassify are marked deprecated; the plan is to remove them and keep only the unrecognized_* catch-all in code.

Classifiers covered

  • K BIZ bill payment / other-bank / PromptPay success → expense, ready if amount and reference are extracted.
  • Status ... (Approved) companion messages → ignored, no Telegram.
  • K SHOP settlement / deposit failures → review.
  • Merchant fee failures → review.
  • OTP / login / admin / security mail → ignored.
  • Statements, e-documents, subscription renewals → review.
  • Enabled email_classification_rules rows match first by priority; built-in defaults apply when no DB rule matches.
  • Financial Ledger side effects require a ready expense + handler_key='company_ledger_expense' + Neon ledger_enabled=true.
  • Everything else → review, with K BIZ-specific unrecognized_kbiz subtype when applicable.
